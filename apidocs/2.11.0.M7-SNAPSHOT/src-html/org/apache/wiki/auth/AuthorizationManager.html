038/**
039 * <p>Manages all access control and authorization; determines what authenticated users are allowed to do.</p>
040 * <p>Privileges in JSPWiki are expressed as Java-standard {@link java.security.Permission} classes. There are two types of permissions:</p>
041 * <ul>
042 *   <li>{@link org.apache.wiki.auth.permissions.WikiPermission} - privileges that apply to an entire wiki instance: <em>e.g.,</em>
043 *   editing user profiles, creating pages, creating groups</li>
044 *   <li>{@link org.apache.wiki.auth.permissions.PagePermission} - privileges that apply to a single wiki page or range of pages:
045 *   <em>e.g.,</em> reading, editing, renaming
046 * </ul>
047 * <p>Calling classes determine whether they are entitled to perform a particular action by constructing the appropriate permission first,
048 * then passing it and the current {@link Session} to the {@link #checkPermission(Session, Permission)} method. If
049 * the session's Subject possesses the permission, the action is allowed.</p>
050 * <p>For WikiPermissions, the decision criteria is relatively simple: the caller either possesses the permission, as granted by the wiki
051 * security policy -- or not.</p>
052 * <p>For PagePermissions, the logic is exactly the same if the page being checked does not have an access control list. However, if the
053 * page does have an ACL, the authorization decision is made based the <em>union</em> of the permissions granted in the ACL and in the
054 * security policy. In other words, the user must be named in the ACL (or belong to a group or role that is named in the ACL) <em>and</em>
055 * be granted (at least) the same permission in the security policy. We do this to prevent a user from gaining more permissions than they
056 * already have, based on the security policy.</p>
057 * <p>See the implementation on {@link #checkPermission(Session, Permission)} method for more information on the authorization logic.</p>
058 *
059 * @since 2.3
060 * @see AuthenticationManager
061 */
062public interface AuthorizationManager {
063

076    /**
077     * Returns <code>true</code> or <code>false</code>, depending on whether a Permission is allowed for the Subject associated with
078     * a supplied Session. The access control algorithm works this way:
079     * <ol>
080     * <li>The {@link org.apache.wiki.auth.acl.Acl} for the page is obtained</li>
081     * <li>The Subject associated with the current {@link org.apache.wiki.api.core.Session} is obtained</li>
082     * <li>If the Subject's Principal set includes the Role Principal that is the administrator group, always allow the Permission</li>
083     * <li>For all permissions, check to see if the Permission is allowed according to the default security policy. If it isn't, deny
084     * the permission and halt further processing.</li>
085     * <li>If there is an Acl, get the list of Principals assigned this Permission in the Acl: these will be role, group or user Principals,
086     * or {@link org.apache.wiki.auth.acl.UnresolvedPrincipal}s (see below). Then iterate through the Subject's Principal set and determine
087     * whether the user (Subject) possesses any one of these specified Roles or Principals.</li>
088     * </ol>
089     * <p>
090     * Note that when iterating through the Acl's list of authorized Principals, it is possible that one or more of the Acl's Principal
091     * entries are of type <code>UnresolvedPrincipal</code>. This means that the last time the ACL was read, the Principal (user, built-in
092     * Role, authorizer Role, or wiki Group) could not be resolved: the Role was not valid, the user wasn't found in the UserDatabase, or
093     * the Group wasn't known to (e.g., cached) in the GroupManager. If an <code>UnresolvedPrincipal</code> is encountered, this method
094     * will attempt to resolve it first <em>before</em> checking to see if the Subject possesses this principal, by calling
095     * {@link #resolvePrincipal(String)}. If the (re-)resolution does not succeed, the access check for the principal will fail by
096     * definition (the Subject should never contain UnresolvedPrincipals).
097     * </p>
098     * <p>
099     * If security not set to JAAS, will return true.
100     * </p>
101     *
102     * @param session the current wiki session
103     * @param permission the Permission being checked
104     * @return the result of the Permission check
105     */
106    boolean checkPermission( Session session, Permission permission );
107
108    /**
109     * <p>Determines if the Subject associated with a supplied Session contains a desired Role or GroupPrincipal. The algorithm
110     * simply checks to see if the Subject possesses the Role or GroupPrincipal it in its Principal set. Note that any user (anonymous,
111     * asserted, authenticated) can possess a built-in role. But a user <em>must</em> be authenticated to possess a role other than one
112     * of the built-in ones. We do this to prevent privilege escalation.</p>
113     * <p>For all other cases, this method returns <code>false</code>.</p>
114     * <p>Note that this method does <em>not</em> consult the external Authorizer or GroupManager; it relies on the Principals that
115     * have been injected into the user's Subject at login time, or after group creation/modification/deletion.</p>
116     *
117     * @param session the current wiki session, which must be non-null. If null, the result of this method always returns <code>false</code>
118     * @param principal the Principal (role or group principal) to look for, which must be non-<code>null</code>. If <code>null</code>,
119     *                  the result of this method always returns <code>false</code>
120     * @return <code>true</code> if the Subject supplied with the WikiContext posesses the Role or GroupPrincipal, <code>false</code> otherwise
121     */
122    default boolean isUserInRole( final Session session, final Principal principal ) {
123        if ( session == null || principal == null || AuthenticationManager.isUserPrincipal( principal ) ) {
124            return false;
125        }
126
127        // Any type of user can possess a built-in role
128        if ( principal instanceof Role && Role.isBuiltInRole( (Role)principal ) ) {
129            return session.hasPrincipal( principal );
130        }
131
132        // Only authenticated users can possess groups or custom roles
133        if ( session.isAuthenticated() && AuthenticationManager.isRolePrincipal( principal ) ) {
134            return session.hasPrincipal( principal );
135        }
136        return false;
137    }

139    /**
140     * Returns the current external {@link Authorizer} in use. This method is guaranteed to return a properly-initialized Authorizer, unless
141     * it could not be initialized. In that case, this method throws a {@link org.apache.wiki.auth.WikiSecurityException}.
142     *
143     * @throws org.apache.wiki.auth.WikiSecurityException if the Authorizer could not be initialized
144     * @return the current Authorizer
145     */
146    Authorizer getAuthorizer() throws WikiSecurityException;
147
148    /**
149     * <p>Determines if the Subject associated with a supplied Session contains a desired user Principal or built-in Role principal,
150     * OR is a member a Group or external Role. The rules are as follows:</p>
151     * <ol>
152     * <li>First, if desired Principal is a Role or GroupPrincipal, delegate to {@link #isUserInRole(Session, Principal)} and
153     * return the result.</li>
154     * <li>Otherwise, we're looking for a user Principal, so iterate through the Principal set and see if any share the same name as the
155     * one we are looking for.</li>
156     * </ol>
157     * <p><em>Note: if the Principal parameter is a user principal, the session must be authenticated in order for the user to "possess it".
158     * Anonymous or asserted sessions will never posseess a named user principal.</em></p>
159     *
160     * @param session the current wiki session, which must be non-null. If null, the result of this method always returns <code>false</code>
161     * @param principal the Principal (role, group, or user principal) to look for, which must be non-null. If null, the result of this
162     *                  method always returns <code>false</code>
163     * @return <code>true</code> if the Subject supplied with the WikiContext posesses the Role, GroupPrincipal or desired
164     *         user Principal, <code>false</code> otherwise
165     */
166    boolean hasRoleOrPrincipal( Session session, Principal principal );
167
168    /**
169     * Checks whether the current user has access to the wiki context, by obtaining the required Permission ({@link WikiContext#requiredPermission()})
170     * and delegating the access check to {@link #checkPermission(Session, Permission)}. If the user is allowed, this method returns
171     * <code>true</code>; <code>false</code> otherwise. If access is allowed, the wiki context will be added to the request as an attribute
172     * with the key name {@link org.apache.wiki.api.core.Context#ATTR_CONTEXT}. Note that this method will automatically redirect the user to
173     * a login or error page, as appropriate, if access fails. This is NOT guaranteed to be default behavior in the future.
174     *
175     * @param context wiki context to check if it is accesible
176     * @param response the http response
177     * @return the result of the access check
178     * @throws IOException In case something goes wrong
179     */
180    default boolean hasAccess( final WikiContext context, final HttpServletResponse response ) throws IOException {
181        return hasAccess( context, response, true );
182    }
183
184    /**
185     * Checks whether the current user has access to the wiki context (and
186     * optionally redirects if not), by obtaining the required Permission ({@link WikiContext#requiredPermission()})
187     * and delegating the access check to {@link #checkPermission(Session, Permission)}.
188     * If the user is allowed, this method returns <code>true</code>;
189     * <code>false</code> otherwise. Also, the wiki context will be added to the request as attribute
190     * with the key name {@link org.apache.wiki.api.core.Context#ATTR_CONTEXT}.
191     *
192     * @param context wiki context to check if it is accesible
193     * @param response The servlet response object
194     * @param redirect If true, makes an automatic redirect to the response
195     * @return the result of the access check
196     * @throws IOException If something goes wrong
197     */
198    boolean hasAccess( final WikiContext context, final HttpServletResponse response, final boolean redirect ) throws IOException;

212     * Do not use this method for normal permission checks; use {@link #checkPermission(Session, Permission)} instead.
213     *
214     * @param principals the Principals to check
215     * @param permission the Permission
216     * @return the result
217     */
218    boolean allowedByLocalPolicy( Principal[] principals, Permission permission );
219
220    /**
221     * Determines whether a Subject possesses a given "static" Permission as defined in the security policy file. This method uses standard
222     * Java 2 security calls to do its work. Note that the current access control context's <code>codeBase</code> is effectively <em>this
223     * class</em>, not that of the caller. Therefore, this method will work best when what matters in the policy is <em>who</em> makes the
224     * permission check, not what the caller's code source is. Internally, this method works by executing <code>Subject.doAsPrivileged</code>
225     * with a privileged action that simply calls {@link AccessController#checkPermission(Permission)}.
226     *
227     * @see AccessController#checkPermission(Permission) . A caught exception (or lack thereof) determines whether the
228     *       privilege is absent (or present).
229     * @param session the Session whose permission status is being queried
230     * @param permission the Permission the Subject must possess
231     * @return <code>true</code> if the Subject possesses the permission, <code>false</code> otherwise
232     */
233    boolean checkStaticPermission( Session session, Permission permission );
234
235    /**
236     * <p>Given a supplied string representing a Principal's name from an Acl, this method resolves the correct type of Principal (role,
237     * group, or user). This method is guaranteed to always return a Principal. The algorithm is straightforward:</p>
238     * <ol>
239     * <li>If the name matches one of the built-in {@link org.apache.wiki.auth.authorize.Role} names, return that built-in Role</li>
240     * <li>If the name matches one supplied by the current {@link org.apache.wiki.auth.Authorizer}, return that Role</li>
241     * <li>If the name matches a group managed by the current {@link org.apache.wiki.auth.authorize.GroupManager}, return that Group</li>
242     * <li>Otherwise, assume that the name represents a user principal. Using the current {@link org.apache.wiki.auth.user.UserDatabase},
243     * find the first user who matches the supplied name by calling {@link org.apache.wiki.auth.user.UserDatabase#find(String)}.</li>
244     * <li>Finally, if a user cannot be found, manufacture and return a generic {@link org.apache.wiki.auth.acl.UnresolvedPrincipal}</li>
245     * </ol>
246     *
247     * @param name the name of the Principal to resolve
248     * @return the fully-resolved Principal
249     */
250    Principal resolvePrincipal( final String name );
